## 一、先认清现实：他们现在能干什么，不能干什么

- **能干的**  
  - 听懂复杂指令 → 生成 Python/脚本  
  - 分析文本、趋势 → 给出赚钱项目的建议  
  - 写抓取代码、写数据分析逻辑  
  - 生成内容（文章、文案、代码、方案）

- **目前很难直接干的**  
  - 操作微信、看朋友消息（需要外接协议/自动化，且容易封号）  
  - 全自动“自己决定做什么生意并执行完毕”（需要你定义流程、把关决策）  
  - 自动跨网站登录、下单、回复（需要授权和流程机器人）

**关键认知**：他们不是“雇员”，而是“超能程序员”+“策略助理”。你得把他们输出的代码和策略，放进一个自动化的管道里跑起来。

## 从专业视角看，缺失了什么（要害问题）

### 1. 缺少“手脚层”的工程定义
你说“让 Codex CLI 执行脚本”，这只是把 Python 进程跑起来，不等于操作 GUI、处理反爬、模拟登录、应对验证码。真正的“手脚”应该是：
- **浏览器自动化层**：Playwright / Puppeteer 集群，配合指纹浏览器、IP 代理池。
- **RPA/桌面自动化层**：如果 OpenClaw 是类似 UI.Vision 或自己封装的 PyAutoGUI/Sikuli 工具，那它需要被抽象成一个个可调用的动作接口（action），让 LLM 输出动作序列而不是直接写死脚本。
- **任务队列与调度器**：不能用 cron 粗放执行，必须用 Celery/RQ/自建 job runner，带重试、超时、死信队列。

原回答只给了“脚本 + 计划任务”，这在小规模、低反爬站还能跑，一旦规模化就崩。

### 2. 没有“自主代理循环”的设计
高级自动化不是“人下指令 → AI 生成代码 → 人执行”，而是：
> **感知 → 规划 → 执行 → 观察结果 → 反思 → 调整 → 继续**

你需要构建一个 Agent Loop：
- 用 Claude CLI 作为规划器和反思器。
- 用 Codex CLI 作为代码生成和执行器（但要在隔离环境执行）。
- Hermes 本地模型做脱敏、去 AI 味、快速分类等低成本任务。
- 所有工具通过统一的“工具注册表”暴露给 LLM，采用 ReAct / Plan-and-Solve 模式。

当前你感觉“不太智能”，正是因为它们没被放进这个闭环里，每次对话都失忆，也没有观察执行结果的能力。

### 3. 混合模型的成本与延迟未优化
- Claude CLI 是云端 API，每次推理有成本和延迟，不适合大批量、高频任务（如实时抓取后逐条分析）。
- Hermes 是本地模型，推理免费但智商有限。
专业做法应该是**分层路由**：轻量任务（分类、摘要改写）走本地 Hermes；复杂推理、策略生成走 Claude。并且要做好失败降级（本地模型挂了切云端廉价款）。

### 4. 安全与稳定性完全没提
让 Codex CLI 执行 LLM 生成的代码是**极度危险**的，必须：
- 强制在 Docker/沙箱中运行，限制网络、磁盘写入。
- 代码执行前做静态扫描（bandit/semgrep），禁止 `rm -rf`、反弹 shell 等。
- 所有 API Key、账号凭据走 secrets manager，绝不写入 prompt。

挣钱项目里如果脚本失误删库或泄露信息，损失远超收益。

## 我如何补强这个架构（直接可实施的方案）

### 架构分层
```
[ 控制面 ] 你 (决策、校准)
     ↕
[ 规划与反思 ] Claude CLI (云端大模型)
     ↕
[ 编排层 ] 自建 Agent 引擎 (Python, 管理工具调用、记忆、状态)
     ↕
[ 工具层 ] 代码执行沙箱 / 浏览器集群 / 企微API / 本地Hermes API
     ↕
[ 数据层 ] SQLite/PostgreSQL + 对象存储 (记录抓取结果、分析日志)
```

### 针对你“不能看微信、不能回信息、不能统计”的工程解法
- **微信**：企业微信 + 官方消息推送 API。让 Claude 生成回复草稿，经你确认后通过企微接口发送。对外可包装成“小助理”服务。
- **数据统计**：把所有需要统计的数据源（Excel、CSV、数据库）放在一个 data 目录，Agent 挂载读取权限。每天自动跑分析脚本，生成 PDF 报告存到共享文件夹或发邮件。
- **信息抓取**：搭建一个轻量爬虫管理平台（如用 Scrapy + scrapyd），把目标网站和规则写成配置文件，由 Claude 生成新 spider 代码，上传到平台自动调度。

### 真正能挣钱的“人机协同”模式
不要追求全自动，要追求**人决策、机器干活**的异步流水线。举一个专业的例子：
1. **市场情报**：Agent 每天抓取海外新品众筹、电商飙升词，清洗后由 Hermes 做初步聚类。
2. **机会提示**：Claude 分析聚类结果，挑出 3 个适合你做、利润空间大的商品/内容主题，推送到你的 Telegram。
3. **你选一个**，回复编号。
4. **Agent 触发生产流水线**：生成商品描述、A+ 文案、小红书笔记、翻译多语言版本，全部保存到你的项目文件夹。
5. **你只需审核发布**。

这个模式下，你的核心价值是决策和品味，AI 是无限产能的执行团队。

## 二、关键步骤拆解

### 1. 打开网页
```json
{ "action": "navigate", "url": "https://www.zhihu.com/hot" }
```
Playwright 会像人一样在真实浏览器中输入网址、等待页面渲染完成（网络请求、JS 执行完毕）。

### 2. 等待目标元素出现
网页可能是动态加载的（比如知乎热榜是通过 JS 异步拉取数据再渲染），如果页面还没加载完就去抓，只能拿到空壳。  
Cephalopod 的 `wait` 或 `extract` 默认会在超时时间内自动等待元素可见，也可以显式等待：
```json
{ "action": "wait", "timeout": 3000 }   // 等待3秒，给异步内容时间
```
更精准的做法是**等待特定选择器出现**，这一点在 Playwright 里通过 `page.wait_for_selector` 实现（目前 Cephalopod 可在 `extract` 时自动等待该选择器）。

### 3. 定位元素：如何“选定”你要的内容
这是核心。你需要告诉 Cephalopod **用哪种方式找到想要的数据**。最常用的是 **CSS 选择器**，它通过网页的标签、类名、ID、属性等特征来定位。

举几个例子：
- 找到所有卡片：`div.hot-list-item`  
- 找到每个问题的标题：`h2.HotList-itemTitle`  
- 找到链接地址：`a.title` 然后取 `href` 属性  
- 找到某个特定表格的第三列：`table.data-table tr > td:nth-child(3)`

**如何获得选择器？**（这是你作为人要教给 AI 的，或者让 AI 自己探索）
- 用浏览器开发者工具（F12）→ 鼠标点击元素 → 右键 “Copy selector” 或 “Copy XPath”。

## 一、为什么视觉方案能抗反爬

传统爬虫（包括 Cephalopod 当前的 `extract`）依赖 DOM 结构和 CSS 选择器，这在反爬面前有三重致命弱点：

| 反爬手段 | 对传统爬虫的影响 | 视觉方案如何免疫 |
|---------|----------------|----------------|
| **动态随机类名/ID** (如 `div.abc123`) | 选择器失效，必须频繁人工修复 | 根本不看类名，只看像素 |
| **关键数据放入图片/Canvas** (如价格图、验证码) | 提取不到文本 | OCR + 视觉语言模型直接读出 |
| **接口签名/加密参数** (如 `_signature`, `X-Bogus`) | 需逆向 JS，成本极高 | 不调接口，只看渲染后的屏幕 |
| **行为检测** (鼠标轨迹、停留时长) | 无头模式易被识别 | 用有头浏览器 + 人类操作节奏 |

视觉方案的本质是：**把“机器读代码”变成了“AI看屏幕”，完全站在用户视角**。服务器发给你什么，你就看什么，不需要逆向任何逻辑。这对于抖音、淘宝、亚马逊等强反爬站点几乎是降维打击。

## 二、视觉方案的代价与适用边界

但它不是银弹，因为它有明确的代价：

**1. 成本高**
- 每次分析一张 2000px 的截图，Claude 视觉 token 消耗可能达到 1000-3000 token。
- 高频率抓取（比如每分钟一次）账单会非常可观。

**2. 精度受截图质量影响**
- 长页面需要分段截图，拼接过程可能出错。
- 字体模糊、遮挡、动效可能导致漏读。

**3. 延迟大**
- 截图传输 + 模型推理总延迟约 3-8 秒，不适合实时竞价类场景。

**适用判断标准**：  
**反爬强度高 + 数据量不大 + 结构不规则 → 适合视觉方案**  
**反爬弱 / 结构规整 / 高频大批量 → 还是用传统选择器 + 缓存策略划算**

## 三、在你的集群中集成本地视觉模型

### 3.1 新增一个服务：**Oculus · 本地视觉引擎** (端口 `:19200`)

参照 Cephalopod 的架构，用 FastAPI 包装模型推理。

**功能 API**：

- `/ocr`：输入截图，返回纯文本。
- `/query`：输入截图 + 自然语言问题（如果加载了 Qwen2-VL），返回答案。

### 3.2 代码骨架（以 Florence-2 为例）

```python
from transformers import AutoProcessor, AutoModelForCausalLM
from PIL import Image
import torch

model_id = "microsoft/Florence-2-large"
model = AutoModelForCausalLM.from_pretrained(model_id, trust_remote_code=True).cuda()
processor = AutoProcessor.from_pretrained(model_id, trust_remote_code=True)

def detect_objects(image: Image.Image):
    prompt = "<OD>"  # 目标检测任务
    inputs = processor(text=prompt, images=image, return_tensors="pt").to("cuda")
    generated_ids = model.generate(**inputs, max_new_tokens=1024)
    return processor.batch_decode(generated_ids, skip_special_tokens=False)[0]
```

FastAPI 接口包装后，挂到 `:19200`，在 Codex 的工具注册表中新增：

```yaml
  - /ocr (POST, image_b64)   => 文字
```

## 二、关键退化点深度剖析：微信操控的精度问题

你最在意的应该是微信操作（占了 90% 场景）。原方案中，Florence-2 可以**检测出每个列表项、按钮的精确边界框**，即使这些控件没有文字（比如头像、语音按钮），也能通过视觉特征识别。

精简方案里 Tier 2 只用 **OCR 文字 + 坐标**。这意味着：
- 如果界面是一个标准的列表（如聊天列表），每行都有文字（昵称、最后一条消息摘要），系统 OCR 可以提取这些文字，并返回每个词的位置。此时只需匹配关键词（如“文件传输助手”）即可精准点击。**这种情况下精度几乎无差别**。
- 但如果你想点击一个**无文字的图标**（比如“+”号、表情按钮、语音通话按钮），OCR 就完全失效，只能降级到 Tier 3（DeepSeek flash）去理解整个截图，再让它给出坐标。这不仅多了一次网络调用，而且由于 DeepSeek 返回的坐标可能不够精确，可能还需要二次修正。**这里能力有明显退化**。

然而，在你的实际业务流中，这类“无文字图标”点击并不是高频动作。绝大部分操作是**发送消息、搜索联系人、读取消息**——都有文字，OCR 足以胜任。

```python
import win32gui
import win32con
import win32clipboard
import pyperclip

def send_keystrokes_bg(hwnd, keys):
    """
    向指定窗口句柄发送组合键（后台）。
    keys: 如 '^f', '{ENTER}', '文件传输助手'
    """
    # 解析组合键，用 PostMessage 发送
    # 简单实现：对于文本，逐个字符发送 WM_CHAR；对于特殊键，发送 WM_KEYDOWN/WM_KEYUP
    # 这里提供简化版，仅支持常用键，复杂情况可调用 send_keystrokes_bg_ex
    for key in keys:
        if key in ('^', '+', '%'):  # 修饰键
            continue
        elif key == '{ENTER}':
            win32gui.PostMessage(hwnd, win32con.WM_KEYDOWN, win32con.VK_RETURN, 0)
            win32gui.PostMessage(hwnd, win32con.WM_KEYUP, win32con.VK_RETURN, 0)
        elif key == '{BACKSPACE}':
            win32gui.PostMessage(hwnd, win32con.WM_KEYDOWN, win32con.VK_BACK, 0)
            win32gui.PostMessage(hwnd, win32con.WM_KEYUP, win32con.VK_BACK, 0)
        else:
            # 普通字符
            win32gui.PostMessage(hwnd, win32con.WM_CHAR, ord(key), 0)

def type_text_bg(hwnd, text):
    """后台输入文本（逐字符发送 WM_CHAR）"""
    for ch in text:
        win32gui.PostMessage(hwnd, win32con.WM_CHAR, ord(ch), 0)

def set_edit_text_bg(edit_hwnd, text):
    """直接通过 UIA 设置编辑框文本（推荐方式，零干扰）"""
    # 如果 edit_hwnd 是 UIA wrapper，直接调用
    try:
        edit_hwnd.set_text(text)
        return True
    except:
        pass
    # 否则用 Windows API 设置
    win32gui.SendMessage(edit_hwnd, win32con.WM_SETTEXT, 0, text)
    return True

def paste_without_clobber(hwnd, text):
    """无污染剪贴板粘贴"""
    # 备份当前剪贴板
    try:
        win32clipboard.OpenClipboard()
        old_data = win32clipboard.GetClipboardData(win32con.CF_TEXT)
        win32clipboard.CloseClipboard()
    except:
        old_data = b''
    # 设置新内容
    pyperclip.copy(text)
    # 发送 Ctrl+V（后台）
    win32gui.PostMessage(hwnd, win32con.WM_KEYDOWN, win32con.VK_CONTROL, 0)
    win32gui.PostMessage(hwnd, win32con.WM_KEYDOWN, ord('V'), 0)
    win32gui.PostMessage(hwnd, win32con.WM_KEYUP, ord('V'), 0)
    win32gui.PostMessage(hwnd, win32con.WM_KEYUP, win32con.VK_CONTROL, 0)
    # 恢复原剪贴板（少量延时等待粘贴完成）
    time.sleep(0.2)
    if old_data:
        win32clipboard.OpenClipboard()
        win32clipboard.EmptyClipboard()
        win32clipboard.SetClipboardData(win32con.CF_TEXT, old_data)
        win32clipboard.CloseClipboard()
```

## 解法一：瞬显窗口截图（当前最可行、改动最小）

**核心逻辑**：让 Octopus 在需要截图 OCR 时，自动执行“**恢复窗口 → 截图 → 立即最小化**”，整个视觉暴露时间控制在 **200-400ms**，用户基本无感。

### 1. 关键 API 调用序列
```python
import win32gui, win32con, time
from PIL import Image
import mss

def capture_minimized_window(hwnd):
    """
    截取窗口的真实内容，即使当前最小化。
    步骤：记录原始状态 → 恢复窗口 → 等待重绘 → 截图 → 复原状态。
    """
    # 1. 记录当前窗口放置状态
    placement = win32gui.GetWindowPlacement(hwnd)
    was_minimized = (placement[1] == win32con.SW_SHOWMINIMIZED)

    # 2. 若最小化，则恢复为之前的状态（通常为正常或最大化）
    if was_minimized:
        win32gui.ShowWindow(hwnd, win32con.SW_RESTORE)
        time.sleep(0.15)  # 给窗口一点时间完成重绘

    # 3. 获取窗口实际区域并截图
    rect = win32gui.GetWindowRect(hwnd)
    left, top, right, bottom = rect
    w, h = right - left, bottom - top
    with mss.mss() as sct:
        img = sct.grab({"left": left, "top": top, "width": w, "height": h})
        pil_img = Image.frombytes("RGB", img.size, img.rgb)

    # 4. 如果原来是最小化，重新最小化回去
    if was_minimized:
        win32gui.ShowWindow(hwnd, win32con.SW_MINIMIZE)

    return pil_img
```

### 2. 集成到 Octopus 的 Tier 2
当需要 OCR 聊天内容时，Octopus 会调用此函数获取截图，整个过程窗口只会在屏幕上闪现一下。只要你不是正盯着任务栏，几乎感觉不到。

## 解法二：后台屏幕外渲染捕获（无任何视觉暴露）

如果想做到**绝对零干扰、完全不闪现**，可以通过 Windows 的 **PrintWindow API** 直接要求窗口在后台绘制到 DC（设备上下文）上。

```python
import win32gui, win32ui, win32con

def capture_via_printwindow(hwnd):
    """
    通过 PrintWindow 后台捕获窗口内容，无需恢复窗口。
    适用于微信这类标准 Win32 窗口（多数自绘应用支持，可尝试）。
    """
    left, top, right, bottom = win32gui.GetWindowRect(hwnd)
    w, h = right - left, bottom - top

    hwnd_dc = win32gui.GetWindowDC(hwnd)
    mfc_dc = win32ui.CreateDCFromHandle(hwnd_dc)
    save_dc = mfc_dc.CreateCompatibleDC()

    bitmap = win32ui.CreateBitmap()
    bitmap.CreateCompatibleBitmap(mfc_dc, w, h)
    save_dc.SelectObject(bitmap)

    # 核心：发送 WM_PRINT 消息，让窗口自己绘制到 DC
    result = win32gui.SendMessage(hwnd, win32con.WM_PRINT, save_dc.GetHandleOutput(), 
                                  win32con.PRF_CLIENT | win32con.PRF_CHILDREN)
    if result != 0:
        # 转换为 PIL 图片
        bmpinfo = bitmap.GetInfo()
        bmpstr = bitmap.GetBitmapBits(True)
        from PIL import Image
        img = Image.frombuffer('RGB', (bmpinfo['bmWidth'], bmpinfo['bmHeight']),
                               bmpstr, 'raw', 'BGRX', 0, 1)
        # 清理
        win32gui.DeleteObject(bitmap.GetHandle())
        save_dc.DeleteDC()
        mfc_dc.DeleteDC()
        win32gui.ReleaseDC(hwnd, hwnd_dc)
        return img
    else:
        # 失败，回退到解法一
        return None
```

**注意**：微信窗口可能不完全响应 WM_PRINT（部分自绘控件可能为黑），此时需要回退到解法一的“瞬显截图”。你可以将 `capture_via_printwindow` 作为 Tier 2-1，失败则自动降级为 Tier 2-2（瞬显）。
