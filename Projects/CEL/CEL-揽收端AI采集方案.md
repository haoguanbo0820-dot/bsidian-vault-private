# 多模态 AI 跨境物流全生命周期信息采集方案

> 2026-06-29 | 技术实现方案 | 配套[[CEL-揽收端AI采集需求文档|需求文档]]

---

## Context

CEL（畅达跨境物流）每天处理 30 万单中俄跨境包裹。当前包裹信息采集依赖光照机（X 光）看形状，人工判断品名/价值/是否违禁品/包装要求。每个环节（揽收→入仓→分拣→组包→出关→境外配送）信息采集不完整、不统一，导致：

- 申报品名与实物不符（清关行开箱发现手机申报为卡片、游戏机申报为耳机）
- 1 卢布假货值每天 5000-6000 单卡关
- 平台罚款累计 1517 万（9 万+ 单）
- 取消率 1.7%（平台均值 0.8%）
- 长春仓单周滞留 2000+ 件

目标：用手机/手持终端 + 多模态 AI 替代光照机，在每个节点拍照上传，自动识别货物品名、价值、是否违禁品、包装是否合规、推荐口岸/班次/时效，实现全生命周期数据闭环。

---

## 核心思路

**不是开发一个新产品，是用现成的多模态 AI 能力 + 现有系统拼装。** 关键技术（Claude/GPT-4V 视觉识别）已成熟，难点在数据规则和流程落地。

## 分模块方案

### 模块 1：手持终端采集（微信小程序）

**为什么是小程序不是 APP：**
- 零安装，扫码即用
- 微信生态内，跟现有工作群无缝衔接
- 开发成本最低

**核心功能：**

| 功能 | 描述 | 用到的能力 |
|------|------|-----------|
| 扫码 | 扫包裹条码/面单号 | 微信原生扫码 API（免费） |
| 拍照 | 拍货物正面/包装/标签 | 微信相机 API |
| 自动上传 | 图片 + 元数据（节点/时间/GPS/操作人）→ 后端 | HTTPS POST |
| 离线模式 | 无网时暂存本地，有网时自动补传 | 微信本地存储 |

**每个节点的采集标准（对标国际标准里程碑）：**

| 节点 | 国际标准对标 | 拍什么 | AI 分析输出 | 时效窗口 | 异常定义 |
|------|------------|--------|------------|---------|---------|
| 揽收 | PUP.01-PUP.04 | 货物外观 + 面单 | 品名、估值、违禁品初筛、申报一致性 | 揽收→入仓 < 4h，超 8h 预警 | AI 置信度 < 70% 标记复核 |
| 入仓 | RCV.01-RCV.04 | 多角度 + 包装 + 尺寸参照 | 包装合规、状态对比、危险品标识 | 入仓→分拣 < 8h，超 12h 预警 | 包装破损拒收/换包装 |
| 分拣 | SCN.01-SCN.03 | 货物 + 标签 | 渠道匹配验证、路由推荐 | 分拣→组包 < 4h | 品类与口岸不匹配自动预警 |
| 组包 | CNS.01-CNS.03 | 大包整体 + 内部货物 | 货物清单核对、封口完整性 | 组包→出关 < 12h | 清单不匹配/超重标记 |
| 出关 | 14/15/16/604 | 报关文件 + 货物实物 | HS 编码确认、违禁品终核、申报对比 | 报关→放行 < 24h，超 48h 预警 | 申报不符拦截/查验留底 |
| 境外配送 | 44/34/80/610/611 | 签收拍照 | 签收确认、失败原因分类 | 清关→首派 < 48h，超 72h 预警 | 派送失败自动通知平台 |

> 详细节点定义、状态码对照表、异常处理流程见[[CEL-揽收端AI采集需求文档#八、国际快递全生命周期路由节点|需求文档第八节]]

### 模块 2：多模态 AI 分析

**技术选型：**

| 任务 | 用哪个 | 为什么 |
|------|--------|--------|
| 物品识别 | Claude Opus 4.7 Vision API | CEL 已接入，高分辨率支持运单/物品细节 |
| 文字 OCR | GPT-4V 或 Claude Vision | 面单/标签/报关文件文字提取 |
| 违禁品判定 | 规则引擎（主力）+ LLM（辅助） | 规则确保准确（俄罗斯海关禁运清单是确定的），LLM 处理模糊/边缘情况 |
| HS 编码归类 | 规则引擎 | HS 编码是固定分类体系，建知识库即可 |
| 包装合规检查 | Claude Vision | 描述包装状态，对比标准 |
| 口岸/班次推荐 | 规则引擎 + 实时状态 | 硬约束（货物类型→可走口岸）+ 软约束（时效/成本）|

**API 调用流程（每个包裹一次分析）：**

```
手机拍照 → 后端接收 → 调用 Claude Vision API：
  输入：[图片] + [申报信息] + [当前节点] + [上下文规则]
  输出：{
    识别品名: "手机壳 × 20",
    材质: "TPU 软胶",
    预估价值: "约 200-400 卢布/个",
    违禁品风险: "低（手机壳，非敏感品类）",
    包装状态: "完好，纸箱封装，无破损",
    建议口岸: "珲春（电子产品专线）",
    建议班次: "珲春-乌苏里 每日 14:00 班次",
    预估时效: "5-7 天",
    申报一致性: "一致 ✓"
  }
→ 写入 TMS，更新状态码
```

**成本估算：**

Claude Opus 4.7 Vision API 图片分析约 $0.005-0.01/张。每天 30 万单如果每单分析一次，约 $1500-3000/天。**实际只需要在关键节点分析（揽收+出关），可降至每天 2 个节点 × 组合拍（多件一起）→ 约 $200-500/天。**

### 模块 3：规则知识库

**这是最难的部分——不是技术，是数据积累：**

| 知识库 | 内容 | 来源 |
|--------|------|------|
| 俄罗斯海关禁运清单 | 品类/材质/成分限制 | 俄海关官网 + CEL 历史踩坑记录 |
| HS 编码映射 | 品名 → HS 编码 | 海关总署公开数据 |
| EAC 认证品类 | 需要 EAC 的商品清单 | 俄技术法规 |
| 包装标准 | 各渠道包装要求 | CEL 运营文档 |
| 口岸能力矩阵 | 各口岸可处理的品类/时效 | CEL 运营经验 |
| 班次时刻表 | 各线路发车时间/到达时间 | CEL 运营数据 |
| 俄罗斯海关估值参考 | 常见品类的合理申报价 | 历史数据 + 市场价格 |

**这个知识库建设本身就是一个持续的工作，需要专人维护。但它是核心竞争力。**

### 模块 4：TMS 集成

CEL 现有 TMS 客户端为 WCF 架构（`net.tcp://47.107.84.253:6011`），存在版本检查阻塞问题。集成方式：

**方案 A（推荐）：独立 AI 分析服务 + TMS 数据写入**
- AI 分析结果通过 API 写入 TMS 数据库（需 TMS 供应商开放写入接口）
- 状态码推送逻辑在 AI 服务中独立处理，不依赖 TMS 客户端

**方案 B（备选）：中间层数据平台**
- 如果 TMS 供应商不开放接口，建立独立的数据平台
- AI 分析结果存储在阿里云（已有 ECS）
- 运营看板独立于 TMS 展示
- TMS 数据通过现有 MITM 代理只读同步

### 模块 5：运营看板

**不是花哨的大屏，是解决实际问题的工具：**

| 功能 | 解决什么问题 |
|------|------------|
| 实时异常预警 | 长春仓滞留 2228 件不再靠周报发现 |
| 罚款风险提示 | 发货前就知道哪些可能被罚 |
| 全链路时效可视化 | 每个包裹在哪、预计多久到 |
| 状态码差异监控 | 自动发现 CEL 推送 vs 平台显示的不一致 |
| 自动化申诉材料 | 针对已产生的罚款，自动生成证据 + 申诉模板 |

### 模块 6：全生命周期路由追踪

#### 6.1 节点覆盖范围

CEL 当前只有 6 个操作节点（揽收/入仓/分拣/组包/出关/境外配送），但国际快递标准追踪需要覆盖 **15-25 个里程碑**。本模块将 CEL 的 6 个节点展开到完整的国际标准追踪链：

```
CEL 操作节点（6 个，采集中执行拍照+AI分析）
        +
数据对接节点（8-12 个，自动获取航司/铁路/清关行数据）
        =
完整全链路追踪（15-25 个里程碑）
```

#### 6.2 需对接的外部追踪数据源

| 数据源 | 对接方式 | 获取的节点 | 优先级 |
|--------|---------|-----------|--------|
| 航司/铁路追踪系统 | API / EDI | 105（离开始发地）、106（抵达目的国） | P1 |
| 中国海关单一窗口 | API 查询 | 14/15/16/604 | P1 |
| 俄罗斯海关 | 清关行 API | 17/18/607/608/609 | P1 |
| 境外末端承运人（CDEK/Boxberry/俄邮） | API | 44/34/80/610/611 | P1 |
| Ozon 平台状态 | API / Webhook | Ozon 状态码回调 | P0 |
| Wildberries 平台状态 | API / Webhook | WB 状态码回调 | P0 |

#### 6.3 异常自动处理规则

| 异常 | 检测条件 | 自动动作 |
|------|---------|---------|
| 节点超时 | 超过时效窗口未到达下一节点 | 推送预警 → 超时翻倍自动升级 |
| 海关查验 | 状态码进入 15/18/608 | **自动调取揽收端 AI 照片** → 生成证据包推送清关行 |
| 申报不符 | AI 识别 ≠ 申报信息 | **自动拦截** → 通知卖家更正 |
| 违禁品命中 | 规则引擎判定违禁品 | **自动拦截** → 生成禁运说明 → 通知退货 |
| 派送失败 | 状态码 611 | 记录失败原因 → 72h 内重新派送 → 超时退回 |
| 包裹破损 | AI 检测破损 | 拍照留证 → 生成索赔材料 → 判断责任方 |

#### 6.4 统一状态码体系

系统内部统一使用以下状态码，同时映射到 Ozon/WB 平台状态：

| 阶段 | 状态码范围 | 示例 |
|------|----------|------|
| 揽收 | PUP.01 - PUP.04 | PUP.03 = 揽收完成（AI 拍照通过） |
| 仓储 | RCV.01 - RCV.04 | RCV.03 = 仓内质检（AI 包装检查完成） |
| 分拣 | SCN.01 - SCN.03 | SCN.02 = 已按口岸分拣 |
| 组包 | CNS.01 - CNS.03 | CNS.02 = 已组包（清单核对完成） |
| 出口清关 | 14/15/16/604 | 16 = 海关放行 |
| 国际运输 | 103/105/106/605/606 | 106 = 抵达目的国 |
| 进口清关 | 17/18/607/608/609/614 | 607 = 清关完成 |
| 末端派送 | 44/34/80/610/611/612/613 | 80 = 已签收（全链路完成） |

> 完整状态码对照表（含顺丰 opCode、eTower Event Code、UPS/DHL 码）见[[CEL-揽收端AI采集需求文档#八、国际快递全生命周期路由节点|需求文档第八节]]

## 分阶段落地

### 第一阶段：核心链路跑通（2-4 周）

目标：一条线路（如 Ozon 华南→珲春→乌苏里）跑通全流程。

- 选 1 个揽收点 + 1 个口岸试点
- 开发小程序 MVP（扫码 + 拍照 + 上传，含 6 个操作节点全部对标国际标准里程碑）
- 接入 Claude Vision API（物品识别 + 合规判断 + 状态码自动更新）
- 搭建规则知识库 MVP（禁运清单 + HS 编码基础版 + 口岸能力矩阵）
- 内部状态码体系对齐 UN/CEFACT 54 标准
- 不做 TMS 集成，先独立运行验证效果

**验证指标：** 品名识别准确率 > 85%、违禁品识别准确率 > 95%、日均处理 500 单、节点间时效数据开始积累

### 第二阶段：全线路覆盖 + 外部追踪对接（4-8 周）

- 扩展到所有揽收点和口岸
- 规则知识库完善（加入 EAC/包装/估值）
- 口岸推荐/班次推荐功能
- 异常预警功能（时效超时、海关查验、包裹破损自动检测）
- TMS 集成（方案 A 或 B 确定后实施）
- **对接外部追踪数据源**：航司/铁路 API（105/106）、清关行 API（17/18/607-609）、末端承运人 API（44/34/80）
- Ozon/WB 平台状态码自动同步

### 第三阶段：智能化（8-12 周）

- 罚款自动申诉（联动揽收端 AI 照片生成证据包）
- 全链路预测（基于历史节点时效数据预测整条线路到达时间）
- 承运人绩效对比（各口岸/航司/末端承运人时效数据排名）
- 客户自助查询（全链路可视化轨迹，每一跳可追溯到 AI 照片）

## 不做的事

- 不做 APP（小程序足够）
- 不做自研模型（Claude/GPT-4V 够用）
- 不替代 TMS（补充不是替代）
- 不追求自动化尺寸测量（AR 精度不够，继续用秤和尺子）
- 不做全自动决策（人审核 AI 结果，尤其是违禁品判定）

## 验证方式

1. 试点线路每天 500 单，对比 AI 识别 vs 人工判断的准确率
2. 跟踪试点线路 2 周的罚款变化
3. 对比试点线路 vs 其他线路的取消率差异
4. 记录每个环节的操作时间（拍照 vs 光照机）
