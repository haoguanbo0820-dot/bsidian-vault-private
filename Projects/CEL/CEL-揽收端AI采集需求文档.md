# CEL 揽收端多模态 AI 采集系统 — 需求文档

> **交技术部门** | 2026-06-29 | 基于 CEL 2026 年 6 月实际运营数据

---

## 一、现状问题

### 1.1 当前揽收流程

```
卖家交货 → 光照机（X光）看形状 → 人工判断品名 → 手工记录 → 发往下一环节
```

### 1.2 光照机的局限

- 只能看形状轮廓，不能识别品类/品牌/材质
- 不能判断货值
- 不能判断是否违禁品（需人工对照禁运清单）
- 不能判断包装是否合规
- **没有结构化数据留存** — 看完了就过了，后续环节不知道揽收时看到了什么

### 1.3 因此导致的实际损失（2026 年 6 月数据）

| 问题 | 数据 | 根本原因 |
|------|------|---------|
| 平台罚款 | **1517 万**（9 万+ 单） | 申报信息不准确，源头在揽收端 |
| 1 卢布假货值 | 每天 **5000-6000 单** | 平台营销活动导致，揽收端未识别 |
| 品名与实物不符 | 反复出现 | 光照机看不出手机还是卡片 |
| 取消率 | CEL **1.7%** vs 平台均值 0.8% | 信息错误导致后续环节出问题 |
| 违禁品误判 | 湿巾被标记违禁品 | 没有统一禁运清单对照 |
| 长春仓滞留 | 单周 **2228 件** | 前端信息不准导致分拨错误 |
| 海关开箱查验差异 | 大量 | 清关时才发现申报错误 |

## 二、目标方案

### 2.1 揽收端改造后流程

```
卖家交货 → 扫码（面单号）→ 手机拍照（3 张）→ AI 自动分析 → 数据写入系统 → 放行/拦截
```

### 2.2 用手机/手持终端替代光照机

揽收人员使用**微信小程序**（无需安装 APP），对每个包裹：

| 步骤 | 操作 | 耗时 |
|------|------|------|
| 1. 扫面单 | 微信原生扫码 | 1 秒 |
| 2. 拍货物正面 | 手机拍照 | 2 秒 |
| 3. 拍包装/标签 | 手机拍照 | 2 秒 |
| 4. AI 分析 | 后端自动完成 | < 3 秒 |
| 5. 确认结果 | 人工看一眼通过 | 1 秒 |

**每单总耗时：约 9 秒**（含拍照和 AI 分析等待）

### 2.3 AI 自动输出以下信息

| 分析项 | 输出内容 | 用到的能力 |
|--------|---------|-----------|
| **品名识别** | 品类、品牌、材质、数量 | 多模态 AI（Claude Vision / GPT-4V）|
| **价值评估** | 预估市场价（卢布/人民币） | AI 识别 + 价格数据库 |
| **违禁品判定** | 是否违禁品、风险等级 | 规则引擎（俄罗斯海关禁运清单）+ AI 辅助 |
| **包装检查** | 包装是否完好、是否符合渠道要求 | AI 视觉检查 |
| **HS 编码** | 自动归类的海关编码 | 规则引擎 |
| **申报一致性** | 与卖家申报品名/价值的对比结果 | AI 交叉比对 |
| **路由推荐** | 建议走哪个口岸、哪个班次 | 规则引擎 + 实时数据 |

## 三、在揽收端解决问题能省多少

### 3.1 时间节省

| 对比维度 | 当前（光照机 + 人工） | 改造后（手机 + AI） | 节省 |
|---------|-------------------|-------------------|------|
| 每单检查时间 | 30-60 秒（光照+判断+记录） | 9 秒（扫码+拍照+AI） | **70-85%** |
| 日均揽收 30 万单人时 | 2500-5000 小时 | 750 小时 | 省 **1750-4250 小时/天** |
| 品名争议事后核查 | 清关环节逐单排查（数小时/天） | 揽收端当场判定 | 省掉**整个事后核查环节** |
| 罚款申诉材料整理 | 人工逐单翻记录（数天/批次） | AI 自动生成证据包 | 从**数天 → 分钟级** |

### 3.2 问题拦截（揽收端拦截率预估）

| 问题类型 | 当前每日发生量 | 揽收端可拦截 | 年化止损 |
|---------|-------------|------------|---------|
| 1 卢布假货值 | 5000-6000 单/天 | **90%**（AI 估值 vs 申报价差异 > 阈值自动拦截） | 避免约 5000 单/天的海关风险 |
| 品名申报不符 | 未知（海关开箱才暴露） | **85%**（AI 识别品名 vs 申报品名不一致自动标记） | 减少海关查验差异 |
| 违禁品误判 | 偶发（如湿巾事件） | **95%**（规则引擎精确匹配禁运清单） | 避免合规事故 |
| 包装不合规 | 未知 | **80%**（AI 视觉检查包装破损/标识缺失） | 减少运输破损 + 平台罚款 |
| 分拨错误（如长春滞留） | 单周 2000+ 件 | **60%**（路由推荐自动匹配正确口岸） | 减少滞留件 |

### 3.3 罚款止损估算

当前累计罚款：**1517 万**（涉及 9 万+ 单）

罚款根因分析：
- 信息申报错误类：约 40-50%
- 时效超时类：约 30-35%
- 操作失误类：约 15-20%

**揽收端改造后，申报错误类罚款可减少 80% 以上。**
保守估算：1517 万 × 40% × 80% = **年止损约 485 万**。

### 3.4 取消率改善

当前 CEL 取消率 1.7%（平台均值 0.8%）→ 目标降至 **1.0% 以下**

每降低 0.1% 取消率 = 日均减少 300 单取消 = 年减少 11 万单取消。

## 四、技术实现要求

### 4.1 微信小程序（采集端）

- 无需安装，扫码即用
- 支持离线采集（无网络时暂存本地，恢复后自动上传）
- 6 个操作节点：揽收 / 入仓 / 分拣 / 组包 / 出关 / 境外配送
- 每个节点：扫码面单 → 拍照 → 上传 → 显示 AI 结果

### 4.2 后端 AI 分析服务

- 部署在阿里云 ECS（已有 `39.105.115.6`）
- 调用 Claude Opus 4.7 Vision API 或多模态模型
- 每个包裹输出结构化 JSON（品名/材质/估值/违禁品/包装/HS编码/一致性）
- 接口响应 < 3 秒

### 4.3 规则知识库

需技术部门提前整理并持续维护：

- 俄罗斯海关禁运清单（品类/材质/成分限制）
- HS 编码映射表（品名 → 编码）
- EAC 认证品类清单
- 各渠道包装标准
- 口岸能力矩阵（哪些品类可以走哪些口岸）
- 班次时刻表（各线路发车/到达时间）
- 常见品类市场估值参考

### 4.4 与 TMS 对接

- 现有 TMS 客户端地址：`net.tcp://47.107.84.253:6011`
- 需确认 TMS 是否支持外部写入接口
- 如不支持，AI 分析结果存入独立数据库，TMS 通过只读方式同步

## 五、不做的事

- 不做独立 APP（微信小程序足够）
- 不自研视觉模型（Claude/GPT-4V 识别能力已达标）
- 不替代 TMS（补充 TMS 缺失的信息采集环节）
- 不用 AR 测尺寸（精度不够，继续用秤和尺子）
- 不做全自动放行（违禁品判定需人工确认）

## 八、国际快递全生命周期路由节点

> 参考标准：UN/CEFACT 54 标准化追踪里程碑、UPU S10 追踪号格式、IATA 航空货运标准、顺丰/DHL/UPS/AliExpress/eTower 状态码体系、Ozon FBP/realFBS 平台物流节点

### 8.1 四大阶段总览

国际快递全生命周期分为四个标准阶段，每个阶段包含明确的里程碑节点：

| 阶段 | 英文 | 核心节点数 | 典型耗时 | 涉及方 |
|------|------|----------|---------|--------|
| **揽收阶段** | Collection / Pickup | 3-5 个 | 0.5-1 天 | 揽收网点、揽收司机 |
| **运输阶段** | Transport / Linehaul | 5-8 个 | 2-7 天 | 集货仓、分拣中心、口岸、航司/铁路 |
| **清关阶段** | Customs Clearance | 4-8 个 | 0.5-3 天 | 海关、清关行、检验检疫 |
| **派送阶段** | Delivery / Last Mile | 3-5 个 | 1-3 天 | 境外承运人、末端派送 |

**中俄线路典型全链路耗时：5-14 天**（陆运 7-14 天，空运 5-8 天，铁路 10-18 天）

### 8.3 CEL 各节点详细采集要求

#### 节点 1：揽收（对标 PUP.01-PUP.04）

| 维度 | 要求 |
|------|------|
| **触发动作** | 卖家交货，揽收员扫码 |
| **必拍照片** | ① 货物正面照（展示品类/品牌）② 面单特写（可 OCR）③ 包装整体照 |
| **AI 输出** | 品名识别、价值评估、违禁品初筛、申报一致性比对、包装状态 |
| **必采元数据** | 面单号、卖家ID、揽收时间戳、揽收点 GPS、揽收员 ID、申报品名/货值 |
| **时效窗口** | 揽收→入仓 应在 **4 小时内** 完成，超 8 小时触发预警 |
| **异常定义** | 无网离线（本地暂存，恢复后自动上传）、AI 识别置信度 < 70%（标记人工复核） |

#### 节点 2：入仓（对标 RCV.01-RCV.04）

| 维度 | 要求 |
|------|------|
| **触发动作** | 货物到达 CEL 集货仓，扫描入库 |
| **必拍照片** | ① 货物 4 面照（展示包装完整性）② 标签/唛头特写 ③ 尺寸参照物（如标尺） |
| **AI 输出** | 包装合规检查（破损/压痕/渗漏）、尺寸估算、与揽收时状态对比、危险品标识识别 |
| **必采元数据** | 入库仓编号、入库时间、入库操作人、货架/储位编码、包装类型分类 |
| **时效窗口** | 入仓→分拣 应在 **8 小时内** 完成，超 12 小时触发预警 |
| **异常定义** | 包装破损（自动标记拒收/换包装）、品名与揽收时不符（触发反查）、禁运品二次确认 |

#### 节点 3：分拣（对标 SCN.01-SCN.03）

| 维度 | 要求 |
|------|------|
| **触发动作** | 包裹进入分拣线 |
| **必拍照片** | ① 货物正面 + 标签（展示目的地标识）② 若有渠道标签拍特写 |
| **AI 输出** | 渠道匹配验证（货物品类 vs 分配口岸是否合理）、路由推荐确认 |
| **必采元数据** | 分拣线编号、分拣时间、目的口岸编码、渠道编码、分拣员 ID |
| **时效窗口** | 分拣→组包 应在 **4 小时内** 完成 |
| **异常定义** | 品类与口岸不匹配（如液体→陆运口岸，自动预警改路由）、分拣错误（扫描目的地与实际不符） |

#### 节点 4：组包（对标 CNS.01-CNS.03）

| 维度 | 要求 |
|------|------|
| **触发动作** | 分拣完成后包裹合并为大包 |
| **必拍照片** | ① 大包整体外观 ② 内部货物排列照 ③ 大包标签/编号特写 |
| **AI 输出** | 大包货物清单核对（包裹数 vs 系统数）、大包封口完整性、大包标签信息 OCR |
| **必采元数据** | 大包编号、内含包裹数、组包时间、组包员 ID、预计发车/起飞时间 |
| **时效窗口** | 组包→出关发运 应在 **12 小时内** 完成 |
| **异常定义** | 货物清单不匹配（数量/品类）、大包封口不规范（拍照标记）、大包超重/超尺寸（预警） |

#### 节点 5：出关（对标 14/15/16/604）

| 维度 | 要求 |
|------|------|
| **触发动作** | 大包到达中国口岸，准备出口报关 |
| **必拍照片** | ① 报关文件（面单/清单/发票）② 货物实物（抽查件）③ 封关/铅封 |
| **AI 输出** | 报关单信息 vs 揽收端 AI 识别结果比对、HS 编码二次确认、违禁品最终核查、申报价值一致性 |
| **必采元数据** | 口岸代码、报关单号、HS 编码、申报价值、放行状态、海关查验标记、出关时间戳 |
| **时效窗口** | 口岸申报→放行 应在 **24 小时内** 完成（陆运口岸一般 4-8 小时），超 48 小时标记为"查验中" |
| **异常定义** | 申报品名与实物差异 > 阈值（拦截重新申报）、海关查验开箱（全程 AI 记录留底）、放行延迟（>24h 自动预警） |

#### 节点 6：境外配送（对标 44/34/80/610/611）

| 维度 | 要求 |
|------|------|
| **触发动作** | 包裹到达俄罗斯派送网点，末端配送 |
| **必拍照片** | ① 签收时包裹状态 ② 若派送失败，拍退回原因（地址错误/拒收/无人） |
| **AI 输出** | 签收确认（AI 自动识别签收状态）、派送失败原因分类、包裹最终状态记录 |
| **必采元数据** | 派送网点代码、派送员 ID、签收时间、签收人姓名（如可获取）、失败原因代码 |
| **时效窗口** | 清关放行→首次派送 应在 **48 小时内**，超 72 小时触发"末端延迟"预警 |
| **异常定义** | 派送失败（自动触发 Ozon/WB 通知）、包裹破损（拍照留证，自动生成索赔材料）、超时未签收（每日提醒） |

### 8.5 异常节点定义与自动处理

| 异常类型 | 判定条件 | 严重级别 | 系统自动动作 | 人工介入 |
|---------|---------|---------|-------------|---------|
| **揽收滞留** | 揽收后 8 小时未入仓 | MEDIUM | 推送提醒给揽收员 + 仓库主管 | 超 24 小时升级 |
| **仓内滞留** | 入仓后 12 小时未分拣 | MEDIUM | 推送提醒给仓库 | 超 24 小时升级 |
| **口岸滞留** | 出关申报后 48 小时未放行 | HIGH | 自动查询海关状态 | 立即通知口岸经理 |
| **海关查验** | 状态码进入 15/18/608 | HIGH | **自动调取揽收端 AI 照片 + 申报记录** 生成查验证据包 | 清关行对照处理 |
| **末端延迟** | 清关放行后 72 小时未派送 | HIGH | 自动通知客户 + 平台 | 联系末端承运人 |
| **派送失败** | 状态码 611 | MEDIUM | 自动记录失败原因 + 触发重新派送流程 | 联系收件人 |
| **包裹破损** | AI 在任何节点检测到破损 | HIGH | **自动拍照留证 + 生成破损报告 + 触发索赔** | 判断责任方 |
| **申报不符** | AI 识别品名 vs 申报差异 > 阈值 | CRITICAL | **拦截包裹 + 通知卖家 + 生成更正申报单** | 人工确认后重新申报 |
| **违禁品命中** | 规则引擎判定为违禁品 | CRITICAL | **拦截包裹 + 通知卖家退货** | 人工确认（防误判） |
| **低价货值** | AI 估值 vs 申报价差异 > 5 倍 | HIGH | 标记"海关风险" + 推荐重新估值 | 运营确认后修改申报 |

### 8.6 Ozon / Wildberries 平台特定路由节点

CEL 同时服务 Ozon 和 Wildberries 两大平台，两平台物流模式差异需在路由节点中体现：

#### Ozon FBP（Fulfillment by Partner）

```
卖家仓库 → CEL 揽收 → CEL 集货仓 → 中国口岸 → 俄罗斯口岸 → Ozon 仓库验收 → Ozon 分拨 → Ozon 末端派送
```

特殊节点：

| 节点 | Ozon FBP 要求 | CEL 采集要求 |
|------|-------------|------------|
| Ozon 仓库验收 | 包裹需贴 Ozon 标准面单（含仓储码） | 揽收端 AI 校验面单格式/条码可读性 |
| 仓储码匹配 | 每个包裹对应 Ozon 仓储 SKU | AI 识别货物 vs 仓储码申报是否一致 |
| 验收扫描 | Ozon 仓库入库扫描 | CEL 需提前推送揽收端 AI 数据给 Ozon |
| 发货批次 | 按 Ozon FBS 批次规则发货 | 组包端按批次规则自动校验 |

#### Ozon realFBS（Real Fulfillment by Seller）

```
卖家仓库 → CEL 揽收 → 中国口岸 → 俄罗斯口岸 → 末端派送（卖家指定/平台指定）
```

realFBS 特点：卖家自己管理库存，Ozon 只负责前端展示，物流完全由 CEL 控制。需要比 FBP 更细粒度的追踪。

#### Wildberries（WB）

```
卖家仓库 → CEL 揽收 → CEL 集货仓 → 中国口岸 → 俄罗斯口岸 → WB 集散中心 → WB 仓库 → WB 末端派送
```

WB 特殊要求：

| 节点 | WB 要求 | CEL 采集要求 |
|------|--------|------------|
| WB 集散中心 | 按 WB 指定的集散中心入仓 | 分拣端按 WB 集散中心编码分拣 |
| WB 仓库 | 按商品类目分仓（服装/电子/家居等） | 组包端按类目分组 |
| WB 面单 | WB 专用条形码格式 | 揽收端校验 WB 面单 |
| 供应链编号 | WB 的 поставка（供应批次号） | 全链路关联追踪 |

#### 平台特殊状态码

Ozon 平台状态码（CEL 需同步推送）：

| Ozon 状态 | 含义 | CEL 对应节点 |
|-----------|------|------------|
| `awaiting_packaging` | 等待打包 | 卖家出货前 |
| `awaiting_deliver` | 等待交货 | 揽收前 |
| `awaiting_registration` | 等待登记 | 揽收进行中 |
| `accepted_from_seller` | 已从卖家接收 | **揽收完成（AI 分析通过）** |
| `arrived_at_sorting_center` | 到达分拣中心 | 入仓完成 |
| `sorting` | 分拣中 | 分拣中 |
| `prepared_for_shipment` | 准备发运 | 组包完成 |
| `sent_for_delivery` | 已发运 | 出关放行 |
| `delivering` | 派送中 | 境外配送中 |
| `delivered` | 已签收 | 签收确认 |

## 九、对技术部门的补充要求

基于上述全生命周期路由节点分析，技术部门在实施时需特别注意：

1. **状态码统一**：系统内部状态码应与 UN/CEFACT 54 标准及 Ozon/WB 平台状态码对齐，避免出现当前 CEL 推送状态码与平台显示不一致的问题
2. **异常自动联动**：当系统检测到异常节点（查验、破损、派送失败），应自动调取该包裹在前端节点的 AI 照片和记录，生成证据包
3. **时效 SLA 监控**：每个节点之间的标准时效窗口应写入系统，超时自动预警
4. **跨境运输跟踪**：除了 CEL 自己的 6 个操作节点，系统应预留接口对接航司/铁路/清关行的追踪数据（状态码 105/106/605/606 等）
5. **平台数据同步**：揽收端 AI 分析结果应实时同步给 Ozon/WB 平台，避免平台侧状态滞后
